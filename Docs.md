# Техническая документация проекта "ИБ-Эффект"

## 1. Обзор проекта
**ИБ-Эффект** — это одностраничное веб-приложение (SPA) для расчета экономической эффективности мероприятий по информационной безопасности. Приложение работает полностью на стороне клиента (Client-Side), не требует серверной части и использует стандартные технологии веб-разработки.

### Стек технологий
- **HTML5**: Семантическая разметка, использование `<template>` для рендеринга списков.
- **CSS3**: CSS Variables (Custom Properties) для темизации, Flexbox/Grid для верстки.
- **JavaScript (ES6+)**: Vanilla JS без фреймворков. Манипуляция DOM, управление состоянием через глобальные переменные.

## 2. Структура файлов
```
├── index.html      # Каркас приложения, навигация, контейнеры для контента, шаблоны (<template>)
├── script.js       # Бизнес-логика, управление состоянием, рендеринг, обработчики событий
└── styles.css      # Стилизация, тема (Dark Mode), переменные CSS
```

## 3. Архитектура приложения
Приложение построено по модели **MVC (Model-View-Controller)** в упрощенном виде:
- **Model**: Глобальные массивы и объекты (`criteria`, `assets`, `risks`, `measures`).
- **View**: HTML-шаблоны внутри `index.html` и функции рендеринга (`render*`).
- **Controller**: Функции обработки действий пользователя (`add*`, `delete*`, `save*`), которые обновляют Model и вызывают перерисовку View.

### Жизненный цикл
1.  **Инициализация (`window.onload`)**: Вызывается `initApp()`, который наполняет массивы демо-данными и запускает первичный рендеринг.
2.  **Навигация**: Переключение вкладок через `showTab(n)`, скрывающее/показывающее блоки `.tab-content`.
3.  **Взаимодействие**: Пользователь вводит данные → вызывается функция изменения состояния → обновляются массивы → вызывается соответствующая `render*` функция → обновляется DOM.

## 4. Модель данных (State)
Состояние приложения хранится в глобальных переменных модуля `script.js`.

### 4.1. Критерии (`criteria`)
Объект конфигурации оценки рисков.
```javascript
{
    formula: "damage * probability * priority", // Строковое описание
    damageMax: 4,       // Максимальный балл ущерба
    probMax: 4,         // Максимальный балл вероятности
    priorityMax: 4,     // Максимальный балл приоритета
    riskAppetite: 12,   // Порог риск-аппетита
    costPerPoint: 50000 // Стоимость 1 балла риска в рублях
}
```

### 4.2. Активы (`assets` Array)
```javascript
{
    id: Number,         // Unique ID (Date.now())
    name: String,       // Название актива
    priority: Number,   // Приоритет (1-4)
    value: Number       // Ценность актива (руб)
}
```

### 4.3. Риски (`risks` Array)
```javascript
{
    id: Number,
    threat: String,         // Описание угрозы
    vulnerability: String,  // Описание уязвимости
    assetId: Number|null,   // Связь с активом
    damage: Number,         // Балл ущерба
    probability: Number,    // Балл вероятности
    priority: Number,       // Балл приоритета

    reduceDamage: Number,   // % снижения ущерба мерой
    reduceProb: Number      // % снижения вероятности мерой
```

### 4.4. Защитные меры (`measures` Array)
```javascript
{
    id: Number,
    name: String,       // Название меры
```

## 5. Бизнес-логика и Алгоритмы
Все расчетные функции вынесены в отдельный блок `// ====================== ФОРМУЛЫ ======================`.

### 5.1. Расчет исходного риска
```javascript
function calculateRisk(damage, probability, priority) {
    return damage * probability * priority;
}
```
*Возвращает балл риска (макс. 64 при стандартных настройках).*

### 5.2. Расчет остаточного риска
```javascript
function calculateResidualRisk(risk, reduceDamage, reduceProb) {
    const newDamage = risk.damage * (1 - reduceDamage / 100);
    const newProb = risk.probability * (1 - reduceProb / 100);
    return Math.max(1, Math.round(newDamage * newProb * risk.priority));
}
```
*Логика:* Применяются коэффициенты снижения к параметрам ущерба и вероятности. `Math.max(1, ...)` гарантирует, что риск не станет нулевым визуально (минимум 1 балл), если только вероятность не обнулена явно (в данной реализации минимальный порог 1).

### 5.3. Монетизация риска
```javascript
function calculateExpectedLoss(riskScore) {
    return riskScore * criteria.costPerPoint;
}
```
*Переводит баллы риска в денежный эквивалент (Ущерб).*

### 5.4. Экономическая эффективность (ROSI)
```javascript
function calculateROSI(effect, cost) {
    return cost > 0 ? Math.round((effect / cost) * 100) : 0;
}
```
*Где `effect` = (Снижение ущерба в рублях) - (Стоимость меры).*

### 5.5. Срок окупаемости
```javascript
function calculatePayback(cost, annualSaving) {
    return annualSaving > 0 ? (cost / annualSaving).toFixed(1) : "∞";
}
```

## 6. Рендеринг и DOM
Приложение использует подход **Client-Side Rendering (CSR)** с ручной манипуляцией DOM.

### 6.1. Шаблоны
В `index.html` определены теги `<template id="...">`. Они содержат разметку карточек элементов (активов, рисков, мер), которая не отображается изначально.
Пример использования в `script.js`:
```javascript
const template = document.getElementById('assetTemplate');
const assetEl = template.content.cloneNode(true);
// Заполнение данных через data-template атрибуты
assetEl.querySelector('[data-template="name"]').textContent = a.name;
```
*Преимущество:* Разделение структуры HTML и логики JS.
*Недостаток:* Поиск элементов через `querySelector` внутри клона может быть затратным при больших списках.

### 6.2. Управление вкладками
Функция `showTab(n)`:
1.  Добавляет класс `.hidden` всем `.tab-content`.
2.  Убирает `.hidden` у целевого контента `#content{n}`.
3.  Обновляет классы `.tab-active` у навигации.
4.  Если открыта вкладка "Экономика" (n=5), вызывается `renderEconomicTab()`.

### 6.3. Реактивность
Приложение не использует реактивный фреймворк (Vue/React). Обновление интерфейса происходит явно через вызов функций `renderDashboard()`, `renderRisks()` и т.д. после каждого изменения данных.
*Важно:* При добавлении меры необходимо вручную вызывать перерисовку всех связанных списков (`renderRisks`, `renderMeasures`, `renderDashboard`), чтобы данные синхронизировались.

## 7. Стилизация (CSS)
В `styles.css` используется архитектура на основе **CSS Variables** (`:root`), что позволяет легко менять тему.

### Ключевые переменные
- `--app-background`: Основной фон (темно-синий).
- `--primary-color`: Акцентный цвет (синий).
- `--card-background`: Фон карточек.
- `--text-primary/secondary`: Цвета текста.

### Темизация
Приложение изначально настроено на **Dark Mode**. Для изменения темы достаточно переопределить переменные в `:root` или добавить класс-модификатор на `body`.

## 8. Ограничения и Технические долги
1.  **Отсутствие персистентности**: Данные хранятся только в оперативной памяти. При перезагрузке страницы (`F5`) все данные сбрасываются к демо-значениям из `initApp()`.
    *   *Рекомендация:* Добавить `localStorage.setItem` при каждом изменении массивов и `localStorage.getItem` при инициализации.
2.  **Валидация ввода**: Используется `prompt()` и базовая проверка `if (!name)`. Отсутствует защита от отрицательных чисел или некорректных типов данных в полях стоимости.
3.  **Связи объектов**: Связь между Риском и Мерой реализуется через `measureId` в объекте риска. Функция `linkMeasure` требует ручного ввода ID риска, что неудобно для пользователя.
    *   *Рекомендация:* Реализовать модальное окно с выбором риска из выпадающего списка при привязке меры.
4.  **Масштабируемость**: Глобальные переменные (`let risks = []`) могут привести к конфликтам имен в будущем.
    *   *Рекомендация:* Инкапсулировать состояние в объект `Store` или использовать классы.
5.  **Обработка ошибок**: Отсутствует глобальный обработчик ошибок (`try/catch`), приложение может "упасть" при некорректных данных в DOM.

## 9. Расширение функционала
Для развития проекта рекомендуется следующие точки расширения:
1.  **Экспорт данных**: Добавить функцию генерации JSON или CSV отчета на основе массивов `risks` и `measures`.
2.  **Графики**: Подключить библиотеку (например, Chart.js) для визуализации снижения рисков на Дашборде.
3.  **Настройка шкал**: Сейчас шкалы (1-4) жестко заданы в логике `Math.random` при создании риска. Необходимо сделать их динамическими на основе `criteria.damageMax`.
4.  **История изменений**: Вести лог изменений рисков для аудита.
